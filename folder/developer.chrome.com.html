Phase 1 — Secret Detection

Scanned root, server/, site/_data/, algolia.js, and cloud-secrets.js for hardcoded credentials (API keys, tokens, private keys, connection strings).

Result: No hardcoded secrets found. Algolia credentials are loaded at runtime via require('dotenv').config() from a .env file generated by cloud-secrets.js (GCP Secret Manager integration). The .env file is excluded from source control via .gitignore but IS included in the App Engine deployment artifact (not listed in .gcloudignore). See Finding DCR-01 for the security implication of this design.

Phase 2 — Dependency Scanning

HIGH
nunjucks 3.2.3 — Autoescape Bypass (GHSA-x77j-w7wf-fjmw)
DCR-DEP-01 | CWE-79 | GHSA-x77j-w7wf-fjmw
Affected Package
nunjucks@3.2.3 (pinned in package.json). Fix available in 3.2.4.
Description
Nunjucks versions below 3.2.4 allow autoescape bypass when two user-controlled parameters appear on the same line in a template. An attacker can inject a backslash \ to escape the enclosing quote of the first parameter and inject raw HTML/script content through the second parameter. However, the primary threat in this codebase is the SSTI finding (DCR-01), which renders this autoescape issue secondary — full template execution is already possible.
Exploitability in Context
The /_render endpoint already provides full Nunjucks template execution for authenticated Google users (see DCR-01). The autoescape bypass is therefore a subset of a larger issue. For unauthenticated attack surfaces where Nunjucks renders content with autoescape enabled, this bug remains relevant.
MEDIUM
markdown-it 12.0.1 — ReDoS via Long Inputs (GHSA-6vfc-qv3f-vr6c)
DCR-DEP-02 | CWE-400 | GHSA-6vfc-qv3f-vr6c
Affected Package
markdown-it@12.0.1 (pinned in package.json). Fix in 12.3.2+.
Vulnerable Pattern
// site/_data/process.js exposes process.env to Nunjucks templates.
// The /_render endpoint processes user body via 'njk, md' (Nunjucks then Markdown).
// If attacker sends body with >50K chars of special whitespace pattern, markdown-it stalls.
const md = require('markdown-it')();
md.render(`x ${' '.repeat(150000)} x  \nx`);  // hangs parser
Exploitability
The /_render endpoint processes user-supplied body content through 'njk, md' (Nunjucks then Markdown), meaning markdown-it processes attacker-controlled content. Any valid Google account can authenticate and submit the crafted payload, causing the Node.js event loop to stall and rendering the site inaccessible (DoS). No timeout is imposed on the Eleventy render call.

Phase 3 — Static Analysis

CRITICAL
Server-Side Template Injection via /_render Endpoint (SSTI + process.env Exfiltration)
DCR-01 | CWE-94 / CWE-200
Location
server/render.js — renderHandler function
site/_data/process.js — global data registration
site/en/preview/layouts/blog-post.njk and doc-post.njk — SSTI vector
.cloudbuild/deploy.yaml — confirms secrets are loaded into .env at deploy time
Vulnerability Description
The POST /_render Express endpoint performs preview rendering for content authors. It:
  1. Authenticates the caller using a Google ID token (any valid Google account is accepted — no domain, group, or allowlist restriction)
  2. Accepts user-supplied fm (YAML front matter) and extracts its content field as the document body
  3. Registers the body as a global Eleventy data value (addGlobalData('body', fm.content))
  4. Renders the preview layout, which contains only {% render body %} — processing the body as a full Nunjucks template followed by Markdown
Separately, site/_data/process.js exports { env: process.env }, making all server-side environment variables available to every Nunjucks template as the global variable process. This data is in scope for the nested {% render body %} call because Eleventy's EleventyRenderPlugin passes this.ctx (the full template context including all global data) into the shortcode.
Attack Proof-of-Concept
POST /_render HTTP/1.1
Host: developer.chrome.com
Content-Type: application/json

{
  "t": "<VALID_GOOGLE_ID_TOKEN>",
  "fm": "---\nlayout: layouts/blog-post.njk\ntitle: test\n---\n{{ process.env | dump }}"
}

# Response body will contain all environment variables including:
# ALGOLIA_API_KEY, ALGOLIA_APP_ID, and any other GCP secrets
# loaded by cloud-secrets.js at deploy time
Secret Exfiltration Chain
# .cloudbuild/deploy.yaml — step 2: cloud-secrets.js fetches ALL secrets from
# GCP Secret Manager and writes them to a .env file on the build machine:
  - name: node:16.14.2
    entrypoint: npm
    args: ['run', 'cloud-secrets']  # writes Algolia keys, etc. to .env

# The .env file is NOT in .gcloudignore — it is bundled into the App Engine deploy.
# At runtime, algolia.js calls require('dotenv').config() loading .env into process.env.
# site/_data/process.js then exposes process.env to all templates:

module.exports = {
  env: process.env,  // ALL env vars — including production secrets — exposed to Nunjucks
};

# In preview layout (site/en/preview/layouts/blog-post.njk):
{% render body %}   // ← 'body' is attacker-controlled Nunjucks template content
Authentication Bypass Analysis
The verifyLogin() function in server/render.js only checks that the Google ID token is valid and that payload['sub'] is truthy (a non-empty user ID). There is no check against a specific email domain (e.g. @google.com), no group membership check, and no IP allowlist. Any person with a free @gmail.com account can authenticate and invoke this endpoint. 
async function verifyLogin(token) {
  const ticket = await authClient.verifyIdToken({
    idToken: token,
    audience: CLIENT_ID,   // 636061184119-c4pof3b538r4ibs8bt6g8qdfth958nc9.apps.googleusercontent.com
  });
  const payload = ticket.getPayload();
  return payload && payload['sub'];  // truthy check only — ANY Google account passes
}
5-Gate Exploitability Verification
  • Gate 1 (Attacker-controlled input): req.body.fm content field maps directly to the body global rendered by {% render body %}
  • Gate 2 (No sanitization): Body is passed as raw string to addGlobalData('body', fm.content), then to renderTemplate(str, 'njk, md', this.ctx) — no escaping or filtering applied
  • Gate 3 (Reachable code path): Public HTTPS endpoint at https://developer.chrome.com/_render, accessible to any Google account holder on the internet
  • Gate 4 (Real impact): process.env is in scope via site/_data/process.js; production secrets (Algolia write key, any GCP Secret Manager secrets) are loaded into process.env at deploy time via .cloudbuild/deploy.yaml running cloud-secrets.js
  • Gate 5 (Not framework-mitigated): Nunjucks {% render body %} executes body as a template engine program, not just variable interpolation — autoescape does not prevent template tag execution
Remediation
Immediate (auth hardening): Restrict verifyLogin() to Google accounts with verified @google.com or @chromium.org email addresses by checking payload['email'] and payload['email_verified']. Consider adding an IAP (Identity-Aware Proxy) layer.

Architecture fix: Remove site/_data/process.js or replace it with a minimal allowlist of safe, non-secret environment flags (e.g., NODE_ENV). Never expose process.env as a template global.

Template sandboxing: Use Nunjucks in sandbox mode or replace {% render body %} with a sanitized Markdown-only render (not Nunjucks) for user-supplied content. Alternatively, render user content in a separate sandboxed process.

Secret management: Do not bundle secrets into the App Engine deployment artifact. Use runtime Secret Manager access with the @google-cloud/secret-manager SDK instead of a build-time .env file.

Phase 4 — Authentication & Authorization

MEDIUM
Overly Broad Authentication on Preview Render Endpoint
DCR-02 | CWE-287
Location
server/render.js:45–58 — verifyLogin() function
Description
The OAuth2 Client ID 636061184119-c4pof3b538r4ibs8bt6g8qdfth958nc9.apps.googleusercontent.com is a public OAuth client. Any person with a Google account can obtain a valid ID token for this audience using the standard OAuth2 implicit or auth-code flow with a browser. The verifyLogin() function accepts any token where payload['sub'] is truthy, which is true for every valid token. The endpoint is intended for internal content authors, but has no mechanism to enforce that.
Combined Impact
This misconfiguration is the prerequisite for the SSTI finding (DCR-01). Without it, DCR-01 would require internal Google credentials. With it, DCR-01 is exploitable by any internet user with a Gmail account.
Remediation
Add email/domain verification: if (!payload['email_verified'] || !payload['email'].endsWith('@google.com')) throw new Error('Unauthorized'). Consider switching to Google IAP which enforces group membership at the infrastructure level without code changes.

Phase 5 — Cryptography

No cryptographic operations identified in the server-side code. Authentication delegates entirely to Google's OAuth2/OIDC infrastructure. No custom crypto, weak algorithms, or predictable tokens found.

Notes
Both workflows use safe triggers. check.yml uses pull_request (sandboxed, no secret access for external PRs). The fkirc/skip-duplicate-actions@master action uses a floating @master reference rather than a pinned SHA — a supply chain risk, though the action has read-only semantics. secrets.PERCY_TOKEN in the Percy step is passed via environment variable, not interpolated directly into shell commands.
LOW
GitHub Action Pinned to Mutable Branch Reference
DCR-CI-01 | CWE-1104
Location
.github/workflows/check.yml:13
Vulnerable Line
uses: fkirc/skip-duplicate-actions@master  # ← mutable branch ref, not commit SHA
Description
The @master reference resolves to whichever commit is currently at the tip of the master branch of fkirc/skip-duplicate-actions. If that repository is compromised or the maintainer pushes a malicious update, the code executes in the CI context for all future pull requests. The other actions use pinned version tags (@v2, @v3) which are mutable but typically treated as stable releases. Pinning to a full commit SHA is the secure practice.
Remediation
Pin to a specific commit SHA: uses: fkirc/skip-duplicate-actions@<full-sha>.
