Server-Side Request Forgery (SSRF) via Unvalidated base_url in Mermaid PNG Rendering
SSRF — Partial-URL / Full-URL

References
CWE-918
Summary
The draw_mermaid_png() function in graph_mermaid.py accepts a caller-supplied base_url parameter that is used without any SSRF validation to construct and execute an outbound HTTP GET request. The URL is formed as {base_url}/img/{mermaid_b64}?type=...&bgColor=... and fetched with requests.get(). If an application surfaces this parameter to users (e.g., in a LangGraph graph-visualization endpoint), an attacker can redirect the server to make requests to internal services, including cloud metadata endpoints (AWS IMDSv1 at 169.254.169.254), internal databases, or other private hosts. The rest of the codebase has a robust validate_safe_url() SSRF guard, but it is never called here. Additionally, the proxies parameter is passed unvalidated to requests.get(), allowing an attacker to control the HTTP proxy.
Data Flow (Source → Sink)
Source: draw_mermaid_png(mermaid_syntax, base_url=<attacker_value>) (caller-controlled parameter) →
Propagation: _render_mermaid_using_api(mermaid_syntax, base_url=base_url) (line 320) →
image_url = f"{base_url}/img/{mermaid_syntax_encoded}?type=...&bgColor=..." (lines 440–443) →
Sink: requests.get(image_url, timeout=10, proxies=proxies) (line 456)

Summary
The Reviver.__call__ method in load.py processes JSON objects with {"lc": 1, "type": "secret", "id": ["ENV_VAR_NAME"]} structure. When the caller has set secrets_from_env=True, it will read any environment variable named in id[0] and return it as a plain string. This means a crafted deserialization payload can silently exfiltrate arbitrary env vars — API keys, database passwords, tokens — from the process environment. While the default is False, many real applications enable this feature to wire serialized chain configs to their secret stores. The documentation warns about this but the warning is buried in docstrings.
Data Flow (Source → Sink)
Source: Attacker-controlled JSON string, e.g.: '{"lc":1,"type":"secret","id":["OPENAI_API_KEY"]}' →
Propagation: loads(text, secrets_from_env=True) → json.loads(text, object_hook=Reviver(secrets_from_env=True)) →
Sink: Reviver.__call__ line 416: if self.secrets_from_env and key in os.environ: return os.environ[key] → returns secret value in plaintext
Vulnerable Code

Steps to Reproduce
  1. Find an application that calls loads() or load() with secrets_from_env=True and processes any portion of attacker-controlled input.
  2. Inject the following JSON payload into the deserialized data stream: {"lc": 1, "type": "secret", "id": ["OPENAI_API_KEY"]}
  3. The return value of loads() (or the field where the secret was embedded) will contain the value of os.environ["OPENAI_API_KEY"].
  4. If the application returns or logs the deserialized value, the secret is exfiltrated. Repeat with any env var name (e.g., DATABASE_URL, AWS_SECRET_ACCESS_KEY).

    payload = f'{{"lc": 1, "type": "secret", "id": ["{target}"]}}'
    leaked = loads(payload, secrets_from_env=True)
    if leaked:
        print(f"Exfiltrated {target}: {leaked}")
Impact

Complete exfiltration of any environment variable in the application process, including API keys (OpenAI, Anthropic, AWS), database connection strings, JWT signing secrets, and OAuth client credentials. In a multi-tenant environment, a single attacker request can harvest all secrets used by the application.

Mitigation

Keep secrets_from_env=False (the default). If env-var secrets are necessary, use a restrictive allowlist via secrets_map instead — explicitly map only the secret names the serialized object legitimately needs. Add a deprecation warning or removal timeline for secrets_from_env.

# SAFE: Use explicit secrets_map instead of secrets_from_env
result = loads(
    payload,
    secrets_map={"OPENAI_API_KEY": os.environ.get("OPENAI_API_KEY")},
    secrets_from_env=False,  # Never enable this for untrusted data
)

Summary
The validate_safe_url() function in _ssrf_protection.py contains a test-mode bypass (lines 198–203). When LANGCHAIN_ENV=local_test, any URL whose hostname starts with "test" and contains "server" is returned immediately — before the cloud metadata endpoint check on line 206. The comment on line 205 states "ALWAYS block cloud metadata endpoints (even with allow_private=True)" but this guarantee is broken by the earlier return. An attacker who can influence the LANGCHAIN_ENV environment variable (through a misconfigured Docker environment, environment variable injection, or a CI/CD pipeline that runs in production mode with test env vars) can submit a URL like http://testserver.169.254.169.254.nip.io/latest/meta-data/ and bypass all SSRF protections.

Sink: Caller proceeds to make HTTP request to cloud metadata endpoint

Impact

If LANGCHAIN_ENV=local_test is set in production (a common misconfiguration in containerized deployments where test scripts are run alongside production), SSRF protection against cloud metadata services is completely bypassed for matching hostnames. Mitigation

Move the cloud metadata check before the test-environment bypass, or explicitly perform the cloud metadata check even in test mode. The "ALWAYS block" comment must be enforced in code, not just documentation.

Summary
The jinja2_formatter() function uses jinja2.sandbox.SandboxedEnvironment, which the code itself documents as a "best-effort approach rather than a guarantee of security." This is accurate: the Jinja2 sandbox has known escape techniques using Python attribute traversal, lipsum, cycler, joiner, namespace globals, and format-string exploits. Many public PoCs exist that achieve RCE through Jinja2's SandboxedEnvironment. Applications that accept user-supplied Jinja2 templates (e.g., via a chat interface that lets users customize prompts with template_format='jinja2') are fully vulnerable to RCE. Deserialization of Jinja2 templates is blocked (VULN-2 cross-reference), but runtime usage is not gated.

Summary
The _load_template() function in prompts/loading.py reads a file path from the deserialized prompt configuration dict (the template_path key). def _load_template(var_name, config, base_dir=None):
    if f"{var_name}_path" in config:
        raw_path = config.pop(f"{var_name}_path")
        template_path = Path(raw_path).resolve()
        # Enforce containment within a safe base directory
        if base_dir:
            base = Path(base_dir).resolve()
            if not str(template_path).startswith(str(base)):
                raise ValueError(f"Template path outside allowed directory: {raw_path}")
        if template_path.suffix == ".txt":
            config[var_name] = template_path.read_text(encoding="utf-8")
    return config

Summary
The grep_search tool in FilesystemFileSearchMiddleware (in libs/langchain_v1) accepts an arbitrary pattern string from the calling agent or user, validates it only by confirming it compiles (re.compile(pattern)), then uses it to scan file contents. Summary
The document indexing API (index() and aindex()) defaults to SHA-1 (key_encoder="sha1") for generating document UIDs. SHA-1 is known to be collision-vulnerable (the SHAttered attack demonstrated practical SHA-1 collisions in 2017). An attacker who can inject documents into the RAG pipeline can craft two documents with the same SHA-1 hash, causing the deduplication logic to treat them as identical. This means a malicious document could suppress a legitimate document from being indexed (or vice versa), poisoning the vector store and causing incorrect or malicious information to be retrieved. The code emits a runtime warning about this but retains SHA-1 as the default.
Data Flow (Source → Sink)
Source: Attacker crafts two documents D1 and D2 with SHA-1(content(D1)) == SHA-1(content(D2)) →
Propagation: index(docs=[D1, D2], ...) → _get_document_with_hash(doc, key_encoder="sha1") → _calculate_hash(content, "sha1") → same UUID for both →
Sink: Deduplication logic (_deduplicate_in_order) drops D2 as a duplicate; D1 is indexed. D2 (the malicious or targeted document) is silently excluded.

    """Hashes a string and returns the corresponding UUID."""

Steps to Reproduce
  1. Using the SHAttered attack framework or similar tooling, construct two distinct text strings D1 and D2 where SHA-1(D1) == SHA-1(D2).
  2. Index both D1 and D2 through the LangChain indexing API.
  3. D2 is silently dropped by the deduplication logic since its computed hash matches D1's.
  4. Queries that should retrieve D2 instead retrieve D1 (or nothing), causing incorrect RAG outputs.

Impact

Cache poisoning in RAG (Retrieval-Augmented Generation) systems. An attacker who can influence documents fed into the indexing pipeline can suppress legitimate documents or cause incorrect documents to be surfaced in retrieval queries. In production LLM applications, this could cause the model to provide factually incorrect or attacker-influenced answers based on the poisoned vector store.

Mitigation

Change the default key_encoder from "sha1" to "blake2b" or "sha256". Both are collision-resistant and already supported in the codebase. The existing warning is not sufficient — the default must be changed.
