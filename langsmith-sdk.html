Denial of Service via Malformed langsmith-trace Header
Denial of Service (Unhandled Exception / Input Validation)
Medium
CONFIRMED — No exception handling around header parsing in TracingMiddleware; any malformed header triggers unhandled ValueError → 500 response
Auth Required
None
CVSS Score

CVSS:3.1/AV:N/AC:L/PR:N/UI:N/S:U/C:N/I:N/A:L
References
CWE-248, CWE-20
Summary
The TracingMiddleware ASGI middleware reads the langsmith-trace HTTP request header and passes it to RunTree.from_headers(), which calls _parse_dotted_order(). That function splits the header value on . and then parses each segment with datetime.strptime() and uuid.UUID(). Neither from_headers nor the middleware wraps this parsing in a try/except. Sending a request with any malformed langsmith-trace value (e.g., INVALID, an empty string, or any value that doesn't conform to the %Y%m%dT%H%M%S%fZ<UUID> format) raises an unhandled ValueError that propagates through the context manager's __enter__ up to the ASGI server, resulting in a 500 Internal Server Error for that request. An attacker can reliably trigger this against any endpoint in any application that uses TracingMiddleware.
Data Flow (Source → Sink)
Source: HTTP request header langsmith-trace (attacker-controlled, no validation at boundary)

Flow:
1. TracingMiddleware.__call__ reads scope["headers"] → checks for b"langsmith-trace" key
2. Calls self._with_headers(parent=headers) which is tracing_context(parent=headers)
3. tracing_context calls _get_parent_run({"parent": headers}) before yield
4. _get_parent_run detects headers is a Mapping, calls RunTree.from_headers(headers)
5. from_headers extracts langsmith_trace string → calls _parse_dotted_order(langsmith_trace.strip())
6. _parse_dotted_order calls datetime.strptime(part[:-36], ...) → raises ValueError for malformed input

Sink: Unhandled exception in tracing_context.__enter__ → ASGI framework returns HTTP 500

def _parse_dotted_order(dotted_order: str) -> list[tuple[datetime, UUID]]:
    """Parse the dotted order string."""
    parts = dotted_order.split(".")
    return [
        (
            # Raises ValueError if part shorter than 36 chars or wrong format
            datetime.strptime(part[:-TIMESTAMP_LENGTH], "%Y%m%dT%H%M%S%fZ"),
            UUID(part[-TIMESTAMP_LENGTH:]),
        )
        for part in parts
    ]
Steps to Reproduce
  1. Deploy any Python web application (FastAPI, Starlette, etc.) that wraps its ASGI app with TracingMiddleware.
  2. Send any HTTP request to any endpoint with a malformed langsmith-trace header. The value just needs to not conform to the %Y%m%dT%H%M%S%fZ<UUID> format.
  3. Observe: the server returns HTTP 500 instead of processing the request normally.
Proof of Concept
curl request
curl -H "langsmith-trace: INVALID" http://target-app.example.com/api/endpoint

# Also works with empty value, whitespace, or any non-conforming format:
curl -H "langsmith-trace: " http://target-app.example.com/api/endpoint
curl -H "langsmith-trace: 2024-01-01" http://target-app.example.com/api/endpoint

# Expected: 200 OK (normal app response)
# Actual:   500 Internal Server Error
Impact

Any attacker who can send HTTP requests to an application using TracingMiddleware can force a 500 response on any endpoint by adding a single malformed header. This disrupts legitimate users' requests. In services where reliability is critical (LLM inference, API gateways), this can be a targeted availability attack.

Mitigation

Wrap the header parsing in TracingMiddleware.__call__ with a try/except block. If the header is malformed, log a warning and fall through to calling the application normally without any tracing context.

# python/langsmith/middleware.py
async def __call__(self, scope: dict, receive, send):
    if scope["type"] == "http" and "headers" in scope:
        headers = dict(scope["headers"])
        if b"langsmith-trace" in headers:
            try:
                with self._with_headers(parent=headers):
                    await self.app(scope, receive, send)
                return
            except Exception:
                import logging
                logging.getLogger(__name__).warning(
                    "Invalid langsmith-trace header; skipping trace context."
                )
                # Fall through to serve the request without tracing
    await self.app(scope, receive, send)

Summary
The TracingMiddleware and the underlying RunTree.from_headers() / _Baggage.from_header() code accepts and processes a baggage HTTP header without any authentication or source validation. The baggage header supports the following sub-keys: langsmith-metadata (arbitrary JSON), langsmith-tags (CSV list), and langsmith-project (project name string). Any unauthenticated HTTP client can set these headers, causing the following effects on all traced runs in the application for that request:
  • Inject arbitrary key-value pairs into trace metadata (e.g., fake user IDs, correlation IDs, environment labels)
  • Add arbitrary tags to traces
  • Reroute traces to a different LangSmith project by overriding the project_name

This is particularly dangerous for applications that use trace metadata for security-relevant decisions, billing attribution, anomaly detection, or audit logging based on LangSmith data.
Data Flow (Source → Sink)
Source: HTTP baggage request header (attacker-controlled)

Flow:
1. TracingMiddleware.__call__ passes the full headers dict to tracing_context(parent=headers)
2. _get_parent_run calls RunTree.from_headers(headers)
3. RunTree.from_headers calls _Baggage.from_headers(headers) which parses the baggage HTTP header
4. _Baggage.from_header parses langsmith-metadata=<JSON>, langsmith-tags=<CSV>, langsmith-project=<string> without validation
5. The parsed metadata/tags/project_name are merged into init_args and used to construct a RunTree
6. All traced functions in the request inherit this injected context, sending tainted data to the LangSmith API

Sink: LangSmith trace data store — persisted metadata, tags, and project routing are attacker-controlled

Steps to Reproduce
  1. Deploy an application using TracingMiddleware with LangSmith tracing enabled.
  2. Send a request with a crafted langsmith-trace header (to trigger distributed tracing context) and a malicious baggage header.
  3. Observe in LangSmith that the trace run contains the injected metadata, injected tags, and/or is routed to the attacker-specified project.
Proof of Concept
Inject arbitrary metadata + reroute trace to attacker's project
# First construct a valid dotted order (timestamp + UUID format)
DOTTED_ORDER="20240101T120000000000Z$(python3 -c 'import uuid; print(str(uuid.uuid4()))')"

# Inject metadata: fake environment label and corrupt billing attribution
# Reroute all traces from this request to "attacker-project"
curl \
  -H "langsmith-trace: ${DOTTED_ORDER}" \
  -H 'baggage: langsmith-metadata=%7B%22user_id%22%3A%22admin%22%2C%22env%22%3A%22production%22%2C%22billing_account%22%3A%22compromised%22%7D,langsmith-project=attacker-project,langsmith-tags=injected%2Cmalicious' \
  http://target-app.example.com/api/chat

# All LangSmith trace spans generated during this request will now:
#   - Have metadata: {"user_id": "admin", "env": "production", "billing_account": "compromised"}
#   - Be tagged with: ["injected", "malicious"]
#   - Be routed to project: "attacker-project" (instead of the app's configured project)
Impact

An attacker can corrupt LangSmith trace data for any request processed by an application using TracingMiddleware. Consequences include: (1) Polluting audit logs and trace data with false information; (2) Rerouting traces to an unauthorized project, potentially leaking sensitive conversation data if the attacker has read access to that project; (3) Falsifying metadata used for cost attribution, user tracking, or anomaly detection based on trace data; (4) Injecting unexpected keys/values into metadata that may affect downstream trace processing or alerting logic.

Mitigation

The distributed tracing headers are a by-design feature, but there are several options to reduce risk:

Option 1 (Recommended for production APIs): Require that the langsmith-trace header be cryptographically signed (HMAC) before accepting it as a trusted distributed trace context. Reject headers without a valid signature.

Option 2 (Defense-in-depth): Add middleware-level allowlisting of which metadata keys are accepted from inbound headers, stripping or rejecting unknown keys.

Option 3 (Minimal): Document clearly that TracingMiddleware should only be used in internal, trusted network environments where inbound headers cannot be forged by external clients (i.e., behind an API gateway that strips unknown headers).

# Option 2: Strip non-allowlisted metadata keys from incoming baggage
ALLOWED_METADATA_KEYS = frozenset({"request_id", "session_id", "correlation_id"})

# In _Baggage.from_header, after parsing metadata:
metadata = {k: v for k, v in metadata.items() if k in ALLOWED_METADATA_KEYS}

Summary
When the environment variable LANGSMITH_FAILED_TRACES_DIR is set, the Client._write_trace_to_fallback_dir() method persists failed trace upload payloads to disk as JSON files for later replay. These JSON envelopes include the complete HTTP request headers (the "headers" key) used during the failed upload — which contain the x-api-key (LangSmith API key) in plaintext. While individual files are created with 0o600 permissions (owner read/write only), the directory itself may have broader permissions, and in containerized deployments, shared volumes, or CI/CD runners, these files could be accessible to other processes or persisted in logs/artifacts. This can result in long-term exposure of the LangSmith API key on disk.
Data Flow (Source → Sink)
Source: Failed LangSmith API upload (network error, 5xx, timeout)

Flow:
1. Background trace thread calls _send_multipart_req() and receives a failure
2. Client._dump_failed_trace(body_fn, headers) is called with the full HTTP headers dict
3. _write_trace_to_fallback_dir(directory, body, endpoint=..., headers=headers) serializes the envelope:
   {"version": 1, "endpoint": "runs/multipart", "headers": {"x-api-key": "<API_KEY>", ...}, "body_base64": "..."}
4. File is written to LANGSMITH_FAILED_TRACES_DIR/trace_<ts>_<hex>.json with chmod 0o600

Sink: Plaintext API key in a file on disk at an operator-specified path

Steps to Reproduce
  1. Set LANGSMITH_FAILED_TRACES_DIR=/tmp/traces and LANGSMITH_API_KEY=lsv2_pt_... in a Python application using the LangSmith client.
  2. Temporarily make the LangSmith API unreachable (e.g., block the endpoint at the firewall), then trigger some traced operations.
  3. Inspect the files in /tmp/traces/: each trace_*.json file contains the full x-api-key header value.
  4. In environments where /tmp or the specified directory is world-readable, any local user can read these files.
Impact

A LangSmith API key exposed in a trace dump file grants full API access under that key's permissions — potentially including reading all traces, datasets, and feedback, modifying or deleting projects, and performing any action the key's owner is authorized for. In containerized environments, shared volumes, or CI/CD systems, the risk of key extraction is higher.

Mitigation

Strip or redact the API key from the headers dict before writing to disk. The file is already marked as "for replay", so the API key should be sourced from the live environment at replay time, not stored in the file.

# python/langsmith/client.py — in _write_trace_to_fallback_dir
# Strip sensitive auth headers before persisting
REDACT_HEADERS = frozenset({"x-api-key", "authorization", "cookie", "x-service-key"})
safe_headers = {k: v for k, v in headers.items() if k.lower() not in REDACT_HEADERS}
envelope = {
    "version": 1,
    "endpoint": endpoint,
    "headers": safe_headers,   # API key NOT stored
    "body_base64": base64.b64encode(body).decode(),
}
